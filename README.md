> Важно про demo-версию на Vercel:
> для публичного статического показа может использоваться frontend demo mode без backend и без реальной БД.
> В таком режиме приложение работает на локальной заглушке с sample-товарами, чтобы можно было открыть список, карточки, редактирование и базовые пользовательские сценарии даже без сервера.
> Это сделано именно для удобной демонстрации интерфейса. Основной локальный сценарий проекта по-прежнему рассчитан на запуск вместе с backend.

## Описание проекта

Нужно было разработать веб-приложение продавца с интегрированным AI-ассистентом, который помогает улучшать карточки объявлений.

Пользовательский сценарий выглядит так:

1. Продавец открывает список своих объявлений.
2. Находит нужную карточку через поиск, фильтры, сортировку и пагинацию.
3. Переходит в просмотр объявления.
4. Открывает форму редактирования.
5. Меняет поля вручную или обращается к AI:
   - улучшить описание;
   - сгенерировать описание с нуля;
   - оценить рыночную цену;
   - задать уточняющий вопрос в чате.
6. Сохраняет изменения через API.

## Бизнес-контекст

Качество текста в объявлении напрямую влияет на отклик пользователей:

- чем понятнее и полнее описание, тем выше шанс быстрой продажи;
- многие продавцы не заполняют описание и характеристики полностью;
- у части объявлений цена выставлена интуитивно, без ориентировки по рынку.

AI-ассистент в этом проекте решает именно эти проблемы:

- помогает быстро улучшить текст без навыков копирайтинга;
- подсказывает разумную цену;
- указывает, какие поля ещё стоит заполнить;
- даёт советы по конкретной карточке через чат.

## Архитектура решения

Проект разделён на две части:

- `frontend` — SPA на React + TypeScript;
- `server` — лёгкий Node.js backend-заглушка с совместимым API.

### Frontend

Frontend отвечает за:

- маршрутизацию;
- отображение списка, карточки и формы;
- работу фильтров, сортировки и пагинации;
- локальные UI-состояния;
- сохранение черновиков;
- работу тёмной темы и локализации;
- обращение к backend и LLM.

### Backend

Backend отвечает за:

- `GET /items`;
- `GET /items/:id`;
- `PUT /items/:id`;
- раздачу изображений из `server/images`;
- хранение seed-данных в JSON-файле.

### Почему здесь есть свой backend

В условии задания предполагается, что backend API уже существует. Но в рабочем окружении не было готового сервера, поэтому для полноценно запускаемого и проверяемого решения добавлен совместимый mock backend с тем же контрактом.

Это даёт два преимущества:

- проект можно поднять локально без внешних зависимостей;
- фронтенд остаётся готовым к подключению реального backend, если он будет предоставлен отдельно.

## AI и интеграция с LLM

### Выбранный вариант

В рамках решения реализован вариант 1 из задания — локальный запуск через `Ollama`.

Это один из официально разрешённых способов и хороший выбор для тестового задания, потому что:

- не требуется внешний API-ключ;
- решение полностью воспроизводимо локально;
- удобно демонстрировать AI-функции офлайн.

### Что умеет AI в проекте

#### 1. Генерация и улучшение описания

AI получает контекст объявления:

- категорию;
- название;
- цену;
- заполненные характеристики;
- текущее описание.

Далее модель:

- создаёт описание с нуля, если поле пустое;
- либо улучшает текущее описание без выдумывания несуществующих характеристик.

Пример идеи промпта:

```ts
const prompt = `
Ты помогаешь улучшать объявления на Авито.
Сохраняй только факты из карточки и не придумывай новые характеристики.
Сделай описание понятным, живым и продающим.
Верни только готовый текст описания.
`
```

#### 2. Оценка рыночной цены

AI получает ту же карточку и просится вернуть структурированный ответ в JSON:

```json
{
  "suggestedPrice": 55000,
  "minPrice": 45000,
  "maxPrice": 65000,
  "confidence": "medium",
  "reasoning": "Краткое объяснение оценки"
}
```

На практике локальные модели иногда возвращают JSON нестрого. Поэтому в проекте реализован fallback-парсинг: если ответ невалиден как JSON, приложение всё равно пытается извлечь цену, диапазон, confidence и текст объяснения.

Это сделано в файле:

- `src/services/llm.ts`

#### 3. Чат с AI

Чат работает в контексте конкретного объявления:

- в модель передаётся карточка товара;
- передаётся ограниченная история сообщений;
- AI отвечает как помощник продавца именно по этому товару.

Пример логики:

```ts
const answer = await askAdAssistant(values, history, question, signal, language)
```

### Безопасность

Для локального режима через Ollama секретные ключи не нужны.

Если переключаться на внешний API, например `Grok / xAI / OpenAI / OpenRouter`, в продакшен-сценарии правильнее вынести AI-вызовы на backend и хранить токен в переменных окружения сервера, а не в браузере.

## Принятые решения

### 1. `needsRevision` вычисляется на клиенте

В задании отдельно указано, что этот признак нужно считать самостоятельно на клиентской стороне. Поэтому frontend анализирует:

- есть ли описание;
- заполнены ли параметры текущей категории.

### 2. Использован mock backend

Так как готовый сервер в workspace отсутствовал, сделан локальный backend с тем же API-контрактом.

### 3. Данные хранятся в JSON-файле

Для целей тестового задания полноценная БД не обязательна. В качестве простого и воспроизводимого решения использован JSON-файл:

- `server/data/items.json`

### 4. UI сделан на собственных переиспользуемых компонентах

Вместо тяжёлой UI-библиотеки использованы собственные компоненты:

- `SelectField`
- `ClearableInput`
- `StatusBanner`
- `Pagination`
- `PlaceholderImage`

Это дало более точный контроль над внешним видом под макеты задания.

### 5. AI-слой изолирован

Вся логика работы с LLM сосредоточена в одном месте:

- `src/services/llm.ts`

Это упрощает замену Ollama на внешний API без переписывания страниц и форм.

## Установка и запуск

### Предварительные требования

- Node.js `20+`
- npm
- Ollama

### Вариант 1. Локальный запуск через Ollama

#### 1. Поднять Ollama

```bash
ollama list
ollama serve
```

Если модели ещё нет:

```bash
ollama pull llama3.1:8b
```

#### 2. Установить зависимости

```bash
npm install
```

#### 3. Запустить клиент и backend вместе

```bash
npm run dev
```

После этого будут доступны:

- frontend: `http://localhost:5173/ads`
- backend: `http://localhost:8080`

### Вариант 2. Раздельный запуск

Только frontend:

```bash
npm run dev:client
```

Только backend:

```bash
npm run server:start
```

### Вариант 3. Через Docker Compose

```bash
docker compose up --build
```

Что поднимется:

- frontend на `http://localhost:5173`
- backend на `http://localhost:8080`

Примечание:

- Ollama не контейнеризован в рамках compose;
- frontend обращается к локальной машине через `host.docker.internal`;
- такой способ удобен именно для локальной проверки и демонстрации.

## Переменные окружения

Создай `.env` на основе `.env.example`.

Пример:

```env
VITE_API_PROXY_TARGET=http://localhost:8080
# VITE_API_URL=http://localhost:8080
VITE_OLLAMA_URL=http://localhost:11434
VITE_OLLAMA_MODEL=llama3.1:8b
VITE_REQUEST_TIMEOUT_MS=20000
```

Пояснение:

- `VITE_API_PROXY_TARGET` — backend для локального Vite proxy;
- `VITE_API_URL` — прямой URL API, если нужен прод/внешний backend;
- `VITE_OLLAMA_URL` — адрес Ollama;
- `VITE_OLLAMA_MODEL` — имя модели;
- `VITE_REQUEST_TIMEOUT_MS` — таймаут запросов.
- `VITE_ENABLE_DEMO_FALLBACK` — разрешает автоматический переход на локальные demo-данные, если backend недоступен;
- `VITE_FORCE_DEMO_MODE` — полностью отключает обращения к backend и всегда использует локальную заглушку.

## Demo mode без backend

Для статического предпросмотра, например на `Vercel`, можно показать приложение даже без backend.

В этом режиме:

- список объявлений открывается на локальных sample-данных;
- доступны страницы списка, просмотра и редактирования;
- изменения сохраняются локально в `localStorage`;
- можно показать маршрутизацию, фильтры, пагинацию и форму;
- AI-функции при этом могут быть недоступны, если не настроен внешний LLM.

Как включить такой режим:

```env
VITE_FORCE_DEMO_MODE=true
```

Если переменная не включена, но backend недоступен, приложение всё равно может автоматически переключиться на demo fallback, если:

```env
VITE_ENABLE_DEMO_FALLBACK=true
```

Это удобно для двух сценариев:

- показать интерфейс на Vercel без отдельного backend;
- дать ревьюеру возможность открыть и проверить основные страницы даже без локального сервера.

## Документация по backend

### Контракт

Backend реализует те же маршруты, что описаны в задании:

- `GET /items`
- `GET /items/:id`
- `PUT /items/:id`

### Хранение данных

Данные лежат в:

- `server/data/items.json`

При первом запуске seed-данные создаются автоматически.

### Изображения

Изображения раздаются из:

- `server/images`

Для товара `el-001` добавлено несколько локальных изображений, чтобы продемонстрировать сценарий галереи:

- первая фотография отображается как основная;
- остальные показываются ниже в виде превью;
- при нехватке ширины работает горизонтальный скролл.

### `GET /items`

Поддерживаются query-параметры:

- `q`
- `limit`
- `skip`
