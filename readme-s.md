- как устроен проект

## Как рассказать проект за 1 минуту

Проект - это личный кабинет продавца с AI-помощником для улучшения объявлений. Я сделал 3 основные страницы:

- `/ads` - список объявлений с поиском, фильтрами, сортировкой, пагинацией и layout-переключателем
- `/ads/:id` - страница просмотра объявления
- `/ads/:id/edit` - форма редактирования с AI-функциями

Архитектурно проект разделён на:

- frontend на React + TypeScript + Vite
- слой сервисов для работы с API и LLM
- mock backend на Node.js для локального запуска
- demo mode без backend, чтобы фронт можно было показать отдельно, например на Vercel

AI используется в трёх местах:

- генерация или улучшение описания
- оценка рыночной цены
- чат по конкретному объявлению

Состояние формы хранится в `react-hook-form`, серверные данные - в `TanStack Query`, глобальные UI-настройки - в `Zustand`. Для защиты от потери данных добавлен автосейв черновика в `localStorage`.

## Поток данных в приложении

### 1. Загрузка списка объявлений

1. Пользователь открывает `/ads`
2. `AdsListPage.tsx` читает query-параметры из URL
3. Страница вызывает `getAllItems()` из `src/services/items.ts`
4. `items.ts` либо идёт в backend `/items`, либо переключается на demo fallback
5. Данные нормализуются и обогащаются флагом `needsRevision`
6. Страница делает клиентскую сортировку, пагинацию и рендерит карточки

### 2. Открытие объявления

1. Пользователь кликает по карточке
2. Роутер открывает `/ads/:id`
3. `AdDetailsPage.tsx` вызывает `getItemById(id)`
4. Данные отображаются в карточке товара
5. Если у объявления несколько фотографий, показывается главная фотография и миниатюры ниже

### 3. Редактирование объявления

1. Пользователь открывает `/ads/:id/edit`
2. `AdEditPage.tsx` загружает объявление и переводит его в форму через `mapItemToFormValues()`
3. `react-hook-form` управляет всей формой
4. `useDraftSync()` автоматически пишет изменения в `localStorage`
5. AI-функции используют текущее состояние формы, а не старые данные
6. После `Сохранить` вызывается `updateItem()`
7. После успеха кэш React Query инвалидируется и пользователь уходит на страницу объявления

## `src/pages/AdsListPage.tsx`

Это одна из самых важных страниц.

Что делает:

- читает состояние страницы из URL
- загружает список объявлений
- синхронизирует поиск, фильтры, layout и сортировку с query-параметрами
- делает клиентскую пагинацию по 10 элементов
- рендерит `AdsToolbar`, `AdsFilters`, `AdCard`, `Pagination`

Почему это сделано так:

- URL является источником истины для списка
- это удобно для перезагрузки страницы, шеринга ссылок и возврата назад
- пользователь не теряет состояние фильтров после refresh

Что здесь важно понимать:

- поиск дебаунсится через `useDeferredValue` и `setTimeout`
- параметры страницы меняются через `useSearchParams`
- данные кэшируются `TanStack Query`

Что можно объяснить как сильное решение:

- разделение на `fetchedItems -> filteredItems -> sortedItems -> pagedItems`
- это делает код читаемым и простым для отладки

Ключевые части:

```tsx
const [searchParams, setSearchParams] = useSearchParams()
const query = searchParams.get('q') ?? ''
const categories = parseCategories(searchParams.get('categories'))
const needsRevisionOnly = searchParams.get('needsRevision') === 'true'
const layout = (searchParams.get('layout') as AdsLayout) === 'list' ? 'list' : 'grid'
const page = Math.max(1, Number(searchParams.get('page') || '1'))
const sortBy = (searchParams.get('sort') as SortOption) || 'createdAt-desc'
```

```tsx
const itemsQuery = useQuery({
  queryKey: ['items', query, categories.join(','), needsRevisionOnly, sortBy],
  queryFn: ({ signal }) =>
    getAllItems(
      {
        query,
        categories,
      },
      signal,
    ),
})
```

Как это объяснять на защите:

- фильтры и поиск не держатся только во внутреннем state, потому что URL лучше подходит для экранов списков
- это решение повышает предсказуемость интерфейса

## `src/pages/AdEditPage.tsx`

Это самый сложный файл в проекте.

Что делает:

- загружает объявление
- инициализирует форму
- синхронизирует черновик
- запускает AI для описания и цены
- ведёт AI-чат
- сохраняет изменения

Что важно знать:

- форма сделана на `react-hook-form`
- валидация идёт через `zod`
- AI-запросы и сохранение умеют отменяться через `AbortController`
- AI использует текущее состояние формы
- сообщения чата хранятся в `localStorage`

Почему я выбрал `react-hook-form + zod`:

- `react-hook-form` хорошо подходит для формы с большим количеством полей и контролируемых компонентов
- `zod` удобно использовать для одной типобезопасной схемы валидации

Ключевые блоки файла:

### Инициализация формы

```tsx
const form = useForm<ItemFormValues>({
  resolver: zodResolver(schema),
  mode: 'onBlur',
  defaultValues: EMPTY_FORM_VALUES,
})
```

### Загрузка данных и заполнение формы

```tsx
useEffect(() => {
  if (!itemQuery.data) {
    return
  }

  const nextValues = mapItemToFormValues(itemQuery.data)
  setBaselineValues(nextValues)
  form.reset(nextValues)
}, [form, itemQuery.data])
```

### Запрос на улучшение описания

```tsx
const suggestion = await improveDescription(
  sanitizeFormValues(form.getValues()),
  controller.signal,
  language,
)
setDescriptionSuggestion(suggestion)
```

### Запрос на оценку цены

```tsx
const suggestion = await estimateMarketPrice(
  sanitizeFormValues(form.getValues()),
  controller.signal,
  language,
)
setPriceSuggestion(suggestion)
```

### Отправка сообщения в AI-чат

```tsx
const answer = await askAdAssistant(
  sanitizeFormValues(form.getValues()),
  nextHistory,
  question,
  controller.signal,
  language,
)
```

### Сохранение

```tsx
await updateItem(id, buildItemUpdateInput(values), controller.signal)
await queryClient.invalidateQueries({ queryKey: ['item', id] })
await queryClient.invalidateQueries({ queryKey: ['items'] })
navigate(`/ads/${id}`)
```

Как объяснять этот файл:

- это orchestration layer для страницы редактирования
- он связывает форму, AI, черновик, загрузку и сохранение
- тяжёлая логика вынесена в утилиты и сервисы, поэтому сам файл большой, но не перегружен случайной логикой

## `src/services/items.ts`

Очень важный файл. Это основной data access layer приложения.

Что делает:

- работает с backend `/items`
- нормализует данные ответа
- валидирует их через `zod`
- умеет переключаться в demo mode
- умеет автоматически падать в demo fallback, если backend недоступен

Почему это сильное решение:

- UI не зависит напрямую от нестабильности backend
- формат ответа валидируется на границе системы
- demo mode делает приложение пригодным для статического показа

Что здесь важно понимать:

- `rawItemSchema` и `detailResponseSchema` защищают UI от кривого API
- `normalizeItem()` приводит данные к единому виду
- `shouldUseDemoFallback()` решает, можно ли перейти на локальную заглушку

Ключевая идея:

```ts
if (env.forceDemoMode) {
  return getDemoItems()
}
```

```ts
if (shouldUseDemoFallback(error)) {
  return applyDemoFilters(getDemoItems(), filters)
}
```

Как объяснять:

- я сделал слой адаптации между UI и backend
- если сервер недоступен, frontend всё равно может демонстрировать основные сценарии

## `src/services/llm.ts`

Это главный AI-файл.

Что делает:

- собирает контекст объявления
- отправляет запрос в Ollama
- генерирует описание
- оценивает цену
- отвечает в AI-чате
- парсит и чистит ответ модели

Что важно сказать:

- у меня три отдельные AI-сценария, но единая транспортная функция `requestOllama()`
- промпты разные под каждую задачу
- в ответе по цене есть отдельный разбор JSON, потому что LLM иногда отвечает неидеально

### 1. Генерация описания

Функция:

```ts
improveDescription(values, signal, language)
```

Что делает:

- если описание пустое, просит придумать новое
- если описание есть, просит улучшить
- запрещает модели выдумывать характеристики
- возвращает только готовый текст

### 2. Оценка рыночной цены

Функция:

```ts
estimateMarketPrice(values, signal, language)
```

Что делает:

- просит вернуть JSON
- если модель присылает кривой JSON, код пытается извлечь нужные поля вручную
- если JSON не распарсился, код всё равно пытается достать цену по тексту

Это важная часть для защиты:

- я не доверяю LLM как строго типизированному API
- поэтому сделал защитный парсер

### 3. AI-чат

Функция:

```ts
askAdAssistant(values, history, question, signal, language)
```

Что делает:

- передаёт в модель карточку объявления
- передаёт последние сообщения чата
- просит давать прикладные советы

Почему это нормально:

- чат не абстрактный, а контекстный
- модель отвечает именно по текущему объявлению

### Общая транспортная функция

```ts
async function requestOllama(prompt: string, language: Language, signal?: AbortSignal)
```

Что важно сказать:

- она инкапсулирует вызов `POST /api/generate`
- все AI-функции работают через неё

## Что говорить про AI

### Почему AI вызывается отдельно от backend

Текущее решение для локальной разработки использует прямой вызов Ollama из frontend:

- это быстро для локальной проверки
- не требует внешнего API-ключа
- позволяет показать полностью работающий AI без дополнительного серверного слоя

Но важно честно понимать ограничение:

- для публичного production-деплоя такое решение неидеально
- в публичной версии правильнее было бы перенести вызовы LLM на backend

### Почему выбрана Ollama

- локально и бесплатно
- не требует внешнего доступа
- удобно для демонстрации

### Как формируется AI-контекст

В `llm.ts` есть функция `serializeAdContext()`, которая собирает:

- категорию
- название
- цену
- заполненные характеристики
- текущее описание

Это делает промпт стабильным и повторяемым.

### Как защищён код от плохих AI-ответов

- для цены есть JSON-parser
- есть fallback-парсинг, если JSON сломан
- есть abort для отмены запросов
- есть обработка пустого ответа

### Что говорить про AI-чат

- чат контекстный
- пользователь задаёт вопрос по конкретной карточке
- в модель отправляется не только вопрос, но и история последних сообщений плюс текущее состояние формы

## Что говорить про demo mode

Это важная часть, если спросят про Vercel или отсутствие backend.

Как работает:

- если backend доступен, frontend использует реальные запросы
- если backend недоступен и включён fallback, frontend переключается на локальные demo-данные
- если включён `VITE_FORCE_DEMO_MODE=true`, всё приложение работает без backend

Что это даёт:

- можно показать все страницы на статическом деплое
- можно пройти базовые сценарии даже без сервера

Что честно сказать:

- это не замена настоящей БД
- это демонстрационный fallback для показа интерфейса и сценариев

## Что говорить про localStorage

В проекте `localStorage` используется в трёх местах:

- черновик формы редактирования
- история AI-чата по конкретному объявлению
- глобальные настройки языка и темы

Почему это нормально:

- это именно клиентские состояния, которым не нужен сервер
- их удобно восстанавливать после refresh

## Что говорить про отмену запросов

ТЗ просит поддержать отмену запросов при переходе между страницами.

У меня это реализовано через:

- `signal` в `TanStack Query`
- `AbortController` в `AdEditPage.tsx` для AI-запросов и сохранения

Зачем это нужно:

- чтобы не было гонок запросов
- чтобы не обновлять уже размонтированный интерфейс
- чтобы при повторном запросе не оставался старый ответ

## Какие вопросы могут задать и как отвечать

## "Почему список управляется через URL, а не только через useState?"

Ответ:

Потому что список - это экран, у которого есть состояние поиска, фильтров, сортировки и пагинации. Если хранить это в URL, то:

- страницу можно перезагрузить без потери состояния
- можно поделиться ссылкой
- кнопка Back/Forward работает предсказуемо

## "Почему ты сделал mock backend?"

Ответ:

Потому что frontend по ТЗ зависит от backend API. Для локальной разработки и демонстрации я сделал совместимый mock backend с тем же контрактом, чтобы пройти все пользовательские сценарии без внешней инфраструктуры.

## "Почему AI не на backend?"

Ответ:

Для локальной демонстрации я выбрал прямой вызов Ollama, потому что это быстрее и не требует внешнего API-ключа. Для публичного production-режима я бы перенёс этот слой на backend.

## "Почему логика ревизии вынесена отдельно?"

Ответ:

Потому что одно и то же правило используется в нескольких местах: список, карточка товара, форма редактирования. Если держать его в одном модуле, поведение везде одинаковое.

## "Почему для формы используется mapItemToFormValues и buildItemUpdateInput?"

Ответ:

Потому что форма и серверный payload - это разные представления данных. Я специально сделал явные преобразования в обе стороны, чтобы страница не занималась ручной сборкой payload.

## "Что происходит, если backend недоступен?"

Ответ:

Если включён fallback, frontend автоматически переключается на demo-данные. Если включён принудительный demo mode, backend вообще не нужен.

## "Как ты избежал гонок запросов в AI?"

Ответ:

Через `AbortController`. Перед новым AI-запросом предыдущий прерывается. Это защищает от ситуаций, когда поздний старый ответ затирает новый.
